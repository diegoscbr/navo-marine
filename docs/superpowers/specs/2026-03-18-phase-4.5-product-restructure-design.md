# Phase 4.5 — Product Restructure & Regatta Management Packages

**Date:** 2026-03-18
**Status:** Approved for implementation
**Depends on:** Phase 4 (Stripe webhook handler) complete

---

## Context

The client has requested two significant changes to the product and pricing structure:

1. **Individual rental pricing**: Change from flat per-event fee ($245) to **$35/day** × (event days + extra days requested by customer).
2. **New product category — Regatta Management**: Three new packages targeting race committees and event organizers, bookable via a new self-serve `/packages` page with custom date selection.

These changes require additive schema migrations and a new UI page. No existing tables are dropped or columns removed.

| Product | Category | Price | Payment Mode | Atlas 2 Units | Tablet | Advance Min |
|---|---|---|---|---|---|---|
| Race Committee Package | regatta_management | $105/day | capture | 0 | ✓ | — |
| R/C Windward Leeward Course Package | regatta_management | $170/day | capture | 5 | ✓ | — |
| RaceSense Management Services | regatta_management | $400/day | **hold** | 0 | — | 90 days |

**RaceSense Management Services** uses a Stripe payment hold (auth, not capture). Additional expenses incurred by the human orchestrator are invoiced separately after the event. ## Schema Changes

All changes are additive. No existing columns are dropped unless noted.

### `units` table
```sql
ALTER TABLE units
  ADD COLUMN unit_type TEXT NOT NULL DEFAULT 'atlas2'
    CHECK (unit_type IN ('atlas2', 'tablet'));
```

Tablet units reference a dedicated `tablet` product row (see seeding section). No change to `product_id NOT NULL` constraint.

### `products` table
```sql
ALTER TABLE products
  ADD COLUMN category TEXT NOT NULL DEFAULT 'individual_rental'
    CHECK (category IN ('individual_rental', 'regatta_management')),
  ADD COLUMN price_per_day_cents INT,
  ADD COLUMN payment_mode TEXT NOT NULL DEFAULT 'capture'
    CHECK (payment_mode IN ('capture', 'hold')),
  ADD COLUMN min_advance_booking_days INT,
  ADD COLUMN atlas2_units_required INT NOT NULL DEFAULT 1,
  ADD COLUMN tablet_required BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN capacity INT NOT NULL DEFAULT 10;
```

**`capacity`**: For individual rentals, defaults to 10 (overridden per-event via `rental_event_products.capacity`). For management packages, capacity reflects concurrent bookings allowed: Race Committee = 3, R/C WL = 2, Management Services = 1 (human-constrained).

Update Atlas 2 existing product:
```sql
UPDATE products
SET
  category = 'individual_rental',
  price_per_day_cents = 3500,
  atlas2_units_required = 1,
  tablet_required = FALSE,
  capacity = 10
WHERE slug = 'atlas-2';
```

### `reservations` table
```sql
-- Add new reservation_type value
ALTER TABLE reservations
  DROP CONSTRAINT IF EXISTS reservations_reservation_type_check,
  ADD CONSTRAINT reservations_reservation_type_check
    CHECK (reservation_type IN ('rental_event', 'rental_custom', 'purchase', 'regatta_package'));

-- Add new status value for payment holds
ALTER TABLE reservations
  DROP CONSTRAINT IF EXISTS reservations_status_check,
  ADD CONSTRAINT reservations_status_check
    CHECK (status IN ('reserved_unpaid', 'reserved_authorized', 'reserved_paid', 'cancelled', 'completed'));

-- Add date/extra_days fields
ALTER TABLE reservations
  ADD COLUMN start_date DATE,
  ADD COLUMN end_date DATE,
  ADD COLUMN extra_days INT NOT NULL DEFAULT 0;
```

**Status semantics:**
- `reserved_unpaid` — Stripe session created, awaiting payment (existing)
- `reserved_authorized` — Management Services hold placed on Stripe (new); not yet captured
- `reserved_paid` — Payment captured (existing)
- `cancelled` — Expired or user-cancelled (existing)
- `completed` — Rental finished (existing)

**Date fields:**
- Individual rental (`rental_event`): `start_date`/`end_date` copied from event at checkout time; `extra_days` is user-input.
- Custom rental (`rental_custom`): `start_date`/`end_date` from the date window; `extra_days` is user-input.
- Regatta package (`regatta_package`): `start_date`/`end_date` are user-input; `extra_days` = 0.
- `unit_id` stays but is nullable for multi-unit reservations.

### `rental_event_products` table
```sql
ALTER TABLE rental_event_products
  ADD COLUMN rental_price_per_day_cents INT;

-- Migrate existing flat price to per-day (Atlas 2 events → $35/day)
UPDATE rental_event_products
SET rental_price_per_day_cents = 3500;
```

The checkout route reads `rental_price_per_day_cents` going forward. The old `rental_price_cents` column is kept in the DB but no longer read by application code after this phase.

### New `reservation_units` table
```sql
CREATE TABLE reservation_units (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  reservation_id UUID NOT NULL REFERENCES reservations(id) ON DELETE CASCADE,
  unit_id UUID NOT NULL REFERENCES units(id),
  assigned_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (reservation_id, unit_id)
);

CREATE INDEX idx_reservation_units_reservation ON reservation_units(reservation_id);
CREATE INDEX idx_reservation_units_unit ON reservation_units(unit_id);
```

Used for multi-unit allocations (R/C WL = 5 Atlas 2 + 1 tablet). Single-unit reservations continue using `reservations.unit_id` for simplicity.

### pg_cron cleanup function update
The existing `expire_unpaid_reservations()` function must be updated to also delete `reservation_units` rows when a reservation is cancelled. Since `reservation_units` has `ON DELETE CASCADE` from `reservations`, this is automatically handled when the reservation row is deleted. However, if the function only updates status (sets `status = 'cancelled'`) without deleting rows, the cascade does not fire. The function must explicitly delete `reservation_units` rows for cancelled reservations, or be updated to hard-delete expired reservation rows.

**Resolution**: Add an explicit DELETE to the cleanup function:
```sql
DELETE FROM reservation_units
WHERE reservation_id IN (
  SELECT id FROM reservations
  WHERE status IN ('reserved_unpaid')
    AND expires_at < NOW()
);
```
This runs before the status update so units are freed correctly.

**`reserved_authorized` (Management Services holds):** These do not have an `expires_at` and are not auto-cancelled by the cron job. They are cancelled only via manual admin action (out of scope for this phase). No cron update needed for this status.

## Pricing Logic

### `daysBetween` utility function
Used in both individual rental and package pricing. Implementation:
```typescript
// Inclusive day count: a 3-day event (Mon–Wed) = 3 days
export function daysBetween(start: string, end: string): number {
  const startMs = new Date(start).getTime()
  const endMs = new Date(end).getTime()
  return Math.floor((endMs - startMs) / 86_400_000) + 1
}
```
Both dates are `YYYY-MM-DD` strings. The `+ 1` makes the count inclusive. This function is shared across the checkout route and the UI pricing preview.

### Individual rental (`rental_event`)
```
event_days = daysBetween(event.start_date, event.end_date)
total_cents = (event_days + extra_days) × rental_price_per_day_cents
```

### Custom rental (`rental_custom`)
```
window_days = daysBetween(window.start_date, window.end_date)
total_cents = (window_days + extra_days) × products.price_per_day_cents
```
Uses `products.price_per_day_cents` (Atlas 2 = $35/day) since `date_window_allocations` has no per-window price override.

### Management packages (`regatta_package`)
```
day_count = daysBetween(start_date, end_date)
total_cents = day_count × product.price_per_day_cents
```

## Availability Logic

### Individual rentals (regatta dropdown)
Individual rental products (`category = 'individual_rental'`) will **never** use `payment_mode = 'hold'`, so `reserved_authorized` is unreachable on this path. The existing `checkEventAvailability` and `checkWindowAvailability` functions (which count `reserved_unpaid` and `reserved_paid`) remain correct and do not need updating.

### Management packages (custom dates)

**Step 1 — Booking count check:**
```sql
SELECT COUNT(*) FROM reservations
WHERE product_id = $product_id
  AND status IN ('reserved_unpaid', 'reserved_authorized', 'reserved_paid')
  AND start_date <= $end_date
  AND end_date >= $start_date;
```
Compare against `products.capacity`.

**Step 2 — Unit availability check (R/C WL and Race Committee):**
```sql
-- Count units of required type already allocated in overlapping reservations
SELECT COUNT(DISTINCT ru.unit_id)
FROM reservation_units ru
JOIN reservations r ON ru.reservation_id = r.id
JOIN units u ON ru.unit_id = u.id
WHERE u.unit_type = $unit_type     -- 'atlas2' or 'tablet'
  AND r.status IN ('reserved_unpaid', 'reserved_authorized', 'reserved_paid')
  AND r.start_date <= $end_date
  AND r.end_date >= $start_date;
```
Compare against total fleet units of that type:
```sql
SELECT COUNT(*) FROM units
WHERE unit_type = $unit_type AND status NOT IN ('damaged', 'lost', 'sold');
```
Note: `retired` is not a valid status in the current schema constraint — do not include it.

If `(total - allocated) < required_count`, return unavailable.

This check runs for both `atlas2_units_required` and `tablet_required` (each independently).

### 3-month advance booking (Management Services only)
Validated server-side in `/api/checkout`:
```typescript
if (product.min_advance_booking_days) {
  // daysBetween is inclusive (+1), so subtract 1 to get the exclusive "days from today" count.
  // Example: today=Mar 18, start=Jun 16 → daysBetween=91, daysUntilStart=90 ✓
  const daysUntilStart = daysBetween(new Date().toISOString().slice(0, 10), start_date) - 1
  if (daysUntilStart < product.min_advance_booking_days) {
    return Response.json({ error: 'Bookings require at least 90 days advance notice' }, { status: 400 })
  }
}
```

**1. Dynamic Stripe line item name:**
```typescript
const productName = product.name  // e.g. "Race Committee Package"
```

**3. Payment hold** (Management Services only):
```typescript
const stripeSessionParams = {
  mode: 'payment',
  // ...
  ...(product.payment_mode === 'hold' && {
    payment_intent_data: { capture_method: 'manual' }
  })
```

**4. Stripe metadata** — include all fields needed for webhook reconciliation:
```typescript
metadata: {
  reservation_type,
  product_id,
  event_id: event_id ?? null,
  date_window_id: date_window_id ?? null,
  start_date: start_date ?? null,
  end_date: end_date ?? null,
```

**5. Multi-unit availability**: For packages with `atlas2_units_required > 0` or `tablet_required = true`, run the unit availability check (see above) before creating the Stripe session. Return 409 if units unavailable.

**6. Reservation status on DB insert:**
- `payment_mode === 'capture'` → status = `'reserved_unpaid'`
- `payment_mode === 'hold'` → status = `'reserved_unpaid'` (hold is placed; moves to `reserved_authorized` via webhook in Phase 4)

**Phase 4 webhook note**: The webhook handler (Phase 4) must handle `payment_mode` from Stripe metadata. For holds, it sets status to `reserved_authorized` instead of `reserved_paid`.

**7. Server-side `extra_days` validation:**
```typescript
if (typeof extra_days !== 'undefined') {
  const parsed = parseInt(String(extra_days), 10)
  if (isNaN(parsed) || parsed < 0 || parsed > 14) {
    return Response.json({ error: 'extra_days must be between 0 and 14' }, { status: 400 })
  }
}
```

**8. Cancel URL for packages:**
```typescript
cancel_url: reservation_type === 'regatta_package'
  ? `${baseUrl}/packages?checkout=cancelled`
  : `${baseUrl}/reserve?checkout=cancelled`
```
